# PeerCaaS 메트릭 및 정산 데이터 관리 전략

이 문서는 PeerCaaS 워커 에이전트에서 발생하는 컨테이너 자원 사용량을 수집, 저장 및 중앙 서버로 전송하여 **정산(Billing)** 및 **모니터링**에 활용하는 기술적 전략을 정의합니다.

## 1. 핵심 목표
- **데이터 무결성**: 정산과 직결된 데이터이므로 단 5초의 사용량도 유실되지 않아야 함.
- **보안성**: 호스트 사용자가 데이터를 조작하거나 훔쳐볼 수 없도록 보호.
- **비용 효율성**: 중앙 DB 비용을 최소화하면서 3개월 이상의 시계열 데이터를 보관.
- **회복 탄력성**: 네트워크 장애나 워커 재시작 후에도 미전송 데이터를 복구하여 전송.

## 2. 전체 아키텍처

## 3. 상세 단계별 전략

### 3.1. 수집 (Collection)
- **주기**: 5초마다 실행 중인 모든 컨테이너의 상태를 수집.
- **항목**:
    - **CPU**: 실시간 점유율 (Core 단위로 환산).
    - **Memory**: 실제 사용 중인 메모리 (MB).
    - **Network**: 누적 TX/RX Bytes (**Counter** 방식).
- **특이사항**: 정산 누락 방지를 위해 트래픽 발생 여부와 상관없이 모든 실행 중인 컨테이너를 기록.

### 3.2. 지역 저장소 (Local Persistence)
- **기술**: CGO-free **SQLite (WAL 모드)** 사용.
- **영속성**: Docker **Named Volume**을 사용하여 컨테이너가 삭제되거나 업데이트되어도 데이터 유지.
- **무결성**: "At-least-once" 전략. 중앙 서버로부터 성공 응답(200 OK)을 받기 전까지는 로컬 DB에서 삭제하지 않음.
- **보안 (향후 계획)**: 호스트 사용자의 조작 방지를 위해 AES-GCM 알고리즘으로 레코드 암호화 및 서명 추가 예정.

### 3.3. 전송 (Shipping)
- **주기**: 1분마다 배치(Batch) 전송하여 네트워크 오버헤드 감소.
- **방식**: **Influx Line Protocol**을 사용하여 VictoriaMetrics의 `/write` 엔드포인트로 전송.
- **재시도**: 워커 시작 시 `PENDING` 상태의 모든 과거 데이터를 먼저 조회하여 즉시 전송 시도.

### 3.4. 중앙 저장소 (Central Storage)
- **저장소**: **VictoriaMetrics (LTS 버전)**.
- **보관 주기**: 기본 3개월 (`-retentionPeriod=3`).
- **효율성**: 시계열 데이터를 고도로 압축하여 저장 공간 및 비용 절감.
- **보안**: **Caddy** 리버스 프록시를 통한 HTTPS 강제 및 **Basic Auth** 인증 계층 적용.

## 4. 정산 신뢰성 확보 방안 (Counter-based Logic)
- **네트워크 유실 대응**: 네트워크 사용량은 누적치(Counter)를 전송함.
- **복구 메커니즘**: 특정 5초 구간의 하트비트가 전송되지 않더라도, 다음 전송되는 누적치(`현재 누적치 - 마지막 기록 누적치`)를 통해 공백 기간의 사용량을 소급 적용 가능.

## 5. 보안 가이드라인
- **통신 보안**: `.dev` 도메인 특성에 따른 HTTPS 강제 적용.
- **접근 제어**: 워커는 전용 ID/PW를 통해서만 중앙 서버에 쓰기 권한을 가짐.
- **키 관리**: 암호화 키(Secret Key)는 허브로부터 런타임에 주입받아 메모리에만 보관하여 디스크 노출 방지.
